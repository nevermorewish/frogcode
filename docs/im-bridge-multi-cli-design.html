IM Bridge 多 CLI 适配层设计

将 frogcode 现有 im-bridge sidecar 扩展为多 CLI 适配层 —— 一套飞书通道，动态切换 Claude Code / OpenClaw 等后端。借鉴 cc-connect 的 Agent 接口模式。

Node Sidecar Feishu Claude Code OpenClaw
背景 架构对比 Agent 接口 文件布局 消息流 配置 迁移步骤 参考映射 流式卡片 多实例 设计决策

1.背景与目标

frogcode 现状

Node sidecar 接收飞书消息后，通过 SSE execute 事件通知 Rust；Rust 端 execute_claude_for_chat() 直接 spawn claude CLI，解析 stream-json，HTTP 回调 sidecar 更新卡片。

问题：协议逻辑散在 Rust，只支持 Claude Code 一种 CLI，加新后端要写 Rust。

nexu 的 OpenClaw 控制

TypeScript 实现：openclaw-process.ts（进程管理）+ openclaw-ws-client.ts（JSON-RPC v3 + Ed25519 握手）+ 配置编译器，约 1500+ 行 TS 代码。

启示：openclaw 本身就是 npm 包，用 TS 控制最自然，可直接搬运。

cc-connect 的 Agent 抽象

Go 项目用统一 Agent 接口抽象 Claude Code / Codex / Cursor / Gemini，每个 CLI 一个实现。项目级路由：agent.type = "claudecode"。

启示：接口驱动的多 CLI 适配是已验证的模式。

本次目标

  • 复用 im-bridge，不新建 openclaw-bridge（避免两套 Feishu SDK 冲突）
  • 在 Node sidecar 内引入 cc-connect 式 Agent 接口，把 Claude 执行从 Rust 搬进 sidecar
  • 新增 OpenClaw adapter，直接复用 nexu 的 TS 代码
  • openclaw 不打包：运行时 PATH 查找 / npx / 用户配置路径
  • Rust 层变薄：只做 sidecar 生命周期 + 配置读写 + SSE 转发，删除 ~300 行执行逻辑
  • 未来加 Codex / Gemini 零 Rust 改动

3.Agent 接口（照搬 cc-connect）

在 src-tauri/sidecar/im-bridge/src/agents/types.ts 定义统一接口：

// src/agents/types.ts
export interface Agent {
  name: 'claudecode' | 'openclaw';
  startSession(opts: StartSessionOpts): Promise<AgentSession>;
  stop(): Promise<void>;
}

export interface AgentSession {
  send(prompt: string, files?: string[]): Promise<void>;
  events(): EventEmitter;      // 或 AsyncIterable<AgentEvent>
  cancel(): Promise<void>;
  currentSessionId(): string | null;
  close(): Promise<void>;
}

export type AgentEvent =
  | { type: 'system'; model?: string }
  | { type: 'text'; delta: string }
  | { type: 'tool_use'; name: string; detail: string }
  | { type: 'tool_result'; ok: boolean }
  | { type: 'result'; costUsd?: number; durationMs?: number;
       totalTokens?: number; contextWindow?: number; error?: string }
  | { type: 'session'; sessionId: string };

关键点：所有 UI 层（卡片渲染、stats、工具列表）只消费 AgentEvent，不再关心底下是哪个 CLI。

两个适配器

适配器实现要点来源
ClaudeCodeAgent spawn('claude', ['--resume', sid?, '--output-format', 'stream-json', '--verbose', '--dangerously-skip-permissions', '-p', prompt])，逐行解析 stream-json，归一化成 AgentEvent 把 im_bridge.rs:363-628 的 JSONL 解析逻辑翻译成 TS（约 100 行）
OpenClawAgent spawn openclaw gateway run（或用户配置的 openclawBin），启动后 WebSocket JSON-RPC 连接，Ed25519 握手，send() → ws.request('session.message', ...)，WS 事件归一化为 AgentEvent 从 nexu 复制 openclaw-process.ts / openclaw-ws-client.ts / device identity / config compiler

OpenClaw 不打包

sidecar 按以下顺序查找 openclaw 二进制：

  1. 配置里显式路径 agents.openclaw.binPath
  2. PATH 里的 openclaw
  3. npx openclaw
  4. 找不到 → emit status 事件 {openclawAvailable: false}，前端提示用户安装

5.消息流（新版）

1. Feishu WS → sidecar dispatcher.receive_v1

2. sidecar: agentManager.handle(chatId, text, files)
     ├─ 若 chat 无活动 session → agentManager 按配置 new Agent(...)
     │    → agent.startSession({ cwd, sessionId: loadSession(chatId) })
     └─ session.send(text, files)

3. 循环 for evt of session.events():
     ├─ cardRenderer.update(chatId, evt)    // 节流 300ms
     │    → larkClient.im.v1.message.patch  // 更新卡片
     └─ if evt.type === 'session' → saveSession(chatId, sessionId)

4. evt.type === 'result' → 最终卡片 + 摘除 typing reaction

Rust 完全不参与消息处理。只做以下事：

  • im_bridge_start → spawn node sidecar
  • im_bridge_stop → kill
  • im_bridge_get/save_config → 读写 ~/.frogcode/im-config.json
  • im_bridge_status → 询问 sidecar /health
  • SSE 转发 im-bridge:* 到前端

7.迁移步骤

按"先零回归、再加新功能"的顺序推进：

  1. 抽离基础设施（无功能变化） sidecar 新建 agents/types.ts、agents/normalizer.ts、card-renderer.ts（从 index.ts 抽出 buildCard）。
    校验：TS 编译通过
  2. 实现 ClaudeCodeAgent 把 im_bridge.rs:363-628 的 spawn + JSONL 解析翻译成 TS，归一化为 AgentEvent。
    校验：独立测试 new ClaudeCodeAgent().startSession().send('hi') 能看到 AgentEvent 流
  3. 接入 AgentManager sidecar index.ts：Feishu receive 不再 bus.emit('execute')，改走 agentManager.handle()，内部订阅 events 驱动 card-renderer。
    校验：端到端飞书对话行为与改造前一致（零回归）
  4. 清理 Rust 层 删除 execute_claude_for_chat()、CardState、post_card_to_sidecar()、SSE execute 分支。
    校验：Rust 编译通过，im_bridge.rs 缩减约 300 行
  5. 移植 OpenClaw 控制代码 sidecar 新建 agents/openclaw/：从 nexu 复制 process manager + WS client + device identity + config compiler。
    校验：独立 smoke — sidecar 能拉起本地 openclaw gateway run 并完成 WS 握手
  6. 实现 OpenClawAgent 把 WS JSON-RPC 事件流归一化为 AgentEvent，实现 Agent 接口注册到 manager。
    校验：切 agentType:"openclaw" 后飞书对话正常
  7. 前端切换 UI 在飞书配置页加 agent 选择下拉，热切换后 sidecar 重启 / 重新 init。
    校验：UI 可切换 CLI，立即生效
  8. 打包脚本 esbuild --external:openclaw 确保 openclaw 不进 bundle；ws / tweetnacl 正常打包。
    校验：bundle 体积不膨胀，release 构建产物正常

9.流式卡片：Rust → TS 迁移

Rust 端全删，但状态机不消失 —— thinking → running → complete 三段式、300ms 节流、stats 累积，全部换地方：

能力原位置（Rust）新位置（TS in sidecar）
卡片状态结构 CardState struct (im_bridge.rs:304-330) card-renderer.ts 内的 state 类型
卡片内容推送 post_card_to_sidecar() HTTP 调用 直接 larkClient.im.v1.message.patch()（本来就在 sidecar 里）
stream-json 解析 Rust JSONL parse → CardState 更新（~250 行） ClaudeCodeAgent 产出 AgentEvent，card-renderer 订阅
300ms 节流 Rust update_interval tick 循环 card-renderer.ts 内部 debounce
sidecar HTTP 端点 POST /feishu-card 被 Rust 回调 删除（跨进程调用变成同进程函数调用）

用户感知：行为完全一致。只是代码从 Rust spawn → JSONL → HTTP 回调 sidecar → Feishu API 简化为 TS agent → AgentEvent → 直接 Feishu API。少一跳 HTTP，少一份 struct 定义，少一个端点。

10.多飞书 × 多 CLI？第一版不做

参考 cc-connect 的约束：

  • cc-connect 一个 Project = 一个 Agent（core/engine.go:133：Engine 只持有一个 agent 实例）
  • 想要 2 个飞书机器人接不同 CLI，cc-connect 必须配 2 个 Project，每个用不同 appId
  • 同一个 appId 的 WS 连接会互踢 —— 飞书层面的物理限制

第一版约束：单实例

一个 frogcode = 一个 im-bridge sidecar = 一个飞书 app = 一个 agentType。与 cc-connect 单 project 完全对等，不做多实例。

未来若需要"同时跑两个飞书机器人接不同 CLI"，有两条路径可选：

方案做法复杂度
A. 多 sidecar 实例 Rust 端持有 Vec<ImBridgeInner>，启动 N 个 sidecar 进程，每个独立端口/配置。对齐 cc-connect 多 Project 模式 中
B. sidecar 内多 Engine 单 sidecar 进程内跑多组 {feishuApp + agent}，共享 Node 进程与 lark SDK 低（纯 TS 改动）

第一版先不支持任何一种，待有明确用户场景后再加（建议走 B）。

11.设计决策（已定）

议题决策说明
Session 映射 platform:channelId:userId 三段式 key 对齐 cc-connect core/message.go:131；同一用户在不同 chat 的会话互不干扰；群聊中不同用户各自独立上下文
OpenClaw 启动方式 sidecar 启动时长驻拉起一个 gateway 多 chat 共享一个 openclaw gateway 进程；减少冷启动；WS 连接复用
OpenClaw 未安装 直接报错，前端提示用户安装 不做隐式降级，避免用户配了 openclaw 却默默走 claudecode 导致行为不一致
配置存放 Rust 管写入；配置文件拆开 ~/.frogcode/im-config.json — 飞书 + agentType（基础）
~/.frogcode/agents/claudecode.json — Claude Code 专属
~/.frogcode/agents/openclaw.json — OpenClaw 专属（binPath、stateDir 等）
~/.frogcode/openclaw/ — openclaw runtime 状态（identity、sessions、device-auth），仿 nexu
多飞书 × 多 CLI 第一版不支持 单 sidecar = 单 feishu app = 单 agentType，与 cc-connect 单 project 对等
CardState / 流式卡片 Rust 端删除，逻辑迁移到 card-renderer.ts 状态机、300ms 节流、stats 累积、/feishu-card 端点全部下沉到 sidecar
