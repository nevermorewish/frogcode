01职责划分

Tauri 客户端（重 — 执行层）

  • spawn Agent CLI（Claude / Codex / Gemini / OpenClaw）
  • 三层通讯：stdio + fd3 遥测 + HTTP Hook（Happy 模式）
  • Agent 配置：二进制路径、模型、权限模式、API Key
  • IM 配置 + 连接：微信 iLink / 飞书 WebSocket
  • Skills 加载 + 执行：本地 YAML 热重载
  • 语音处理：STT / TTS（调用远程 API 或本地 Edge）
  • 桌面 GUI：React UI + Tauri IPC
  • 上报状态：定时向 Server 同步客户端状态

Server（轻 — 管控层）

  • 用户认证：JWT 登录、角色（Admin / User）
  • 客户端注册 + 状态聚合：在线/离线、活跃会话
  • 管理员面板：Web UI 管理所有客户端
  • Memory 存储 + 检索：集中式长期记忆
  • 配额 + 用量统计：Token、费用、审计日志
  • 用户 Web 面板：浏览器查看自己客户端状态
  • 不 spawn 任何 Agent
  • 不持有 IM 连接

一句话总结

客户端是手脚（spawn Agent、连 IM、执行 Skill），Server 是大脑（记忆、管控、统计）。客户端可以离线独立工作；Server 提供跨客户端的记忆共享和管理员视野。

04Claude Code 连接（Happy 模式）

客户端本地 spawn Claude Code 进程，通过三层通讯管理。参考 Happy 的 spawn + HTTP Hook + fd3 架构。

通道 1: stdio 管道
stdin ← 用户消息写入
stdout → Agent 输出逐行读取
--output-format stream-json
通道 2: fd3 遥测
拦截 Claude 内部 fetch
fetch-start / fetch-end
驱动 UI "思考中…" 状态
通道 3: HTTP Hook
客户端启动 per-session 端口
POST /hook/session-start
获取 session_id + transcript 
// 客户端 Agent Supervisor 内部伪码

async fn spawn_claude(config: &AgentConfig, prompt: &str, memory_ctx: &str) {
    // 1. 生成 Hook 配置文件
    let hook_port = random_port();
    let hook_settings = generate_hook_settings(hook_port);

    // 2. 构建启动参数
    let args = vec![
        "--session-id", &uuid(),
        "--model", &config.model,
        "--permission-mode", &config.permission_mode,
        "--settings", &hook_settings.path,
        "--append-system-prompt", memory_ctx,  // Memory 注入
    ];

    // 3. spawn 进程（stdio piped + fd3 piped）
    let child = Command::new(&config.binary)
        .args(&args)
        .stdin(Stdio::piped())
        .stdout(Stdio::piped())
        .stderr(Stdio::piped())
        .fd3(Stdio::piped())
        .current_dir(&project_path)
        .env_remove_claude_vars()
        .spawn()?;

    // 4. 启动三个并行读取 task
    tokio::join!(
        read_stdout(child.stdout, event_tx.clone()),   // 通道1
        read_fd3(child.fd3, event_tx.clone()),          // 通道2
        listen_hook(hook_port, event_tx.clone()),        // 通道3
    );

    // 5. 事件汇聚后：
    //   → Tauri emit 到前端 UI
    //   → 转发到 IM 通道
    //   → 上报 Server（状态 + 用量）
    //   → 提取记忆 → 写入 Server Memory
}

管理员对客户端的操作

管理员通过 Web 面板查看状态，但操作指令通过 Server → WebSocket → 客户端下发执行：

  • 请求停止 — Server 发 stop_agent 指令 → 客户端执行 kill
  • 请求重启 — Server 发 restart_agent → 客户端重新 spawn
  • 冻结客户端 — Server 拒绝该客户端的新会话请求，标记为 Frozen
  • 修改配额 — Server 直接更新配额 → 下次客户端上报用量时生效

08Memory 系统

Memory 存储在 Server 上，按用户隔离。客户端在每次对话前向 Server 请求相关记忆，对话后提取新记忆写回 Server。

8.1 四种记忆类型

user — 用户画像
  • 角色、技术栈、偏好、工作方式
  • 例："Rust 高级开发者，偏好零成本抽象"
feedback — 行为反馈
  • 用户纠正或确认的做法 + Why + How to apply
  • 例："不 mock 数据库，因为上次导致线上事故"
project — 项目上下文
  • 目标、决策、约束、截止日期
  • 例："v2.0 auth 重构，合规驱动，截止 4/15"
reference — 外部资源
  • 外部系统中信息的指针
  • 例："Bug 跟踪在 Linear INGEST 项目"

8.2 数据模型

struct MemoryEntry {
    id:            Uuid,
    user_id:       UserId,          // 按用户隔离
    memory_type:   MemoryType,      // User | Feedback | Project | Reference
    title:         String,          // 简短标题
    content:       String,          // Markdown 内容
    embedding:     Vec<f32>,        // 向量嵌入
    relevance:     f32,             // 相关性权重（随时间衰减）
    access_count:  u32,             // 被召回次数
    source:        MemorySource,    // Auto | Manual | IM
    source_client: Option<String>,  // 产生这条记忆的客户端名称
    created_at:    DateTime,
    updated_at:    DateTime,
}

8.4 记忆衰减与维护

机制说明
时间衰减每天 Server 后台 task 将长期未访问的记忆 relevance × 0.98，低于 0.1 标记归档
召回提升每次被召回 relevance += 0.1（上限 1.0），access_count++
冲突替换新记忆与旧记忆语义相似度 > 0.9 时，合并更新而非新增
容量淘汰超过用户配额时，归档 relevance 最低的条目
手动管理用户通过 Web UI 或桌面 GUI 查看、编辑、删除记忆

11.2 离线能力

  • 客户端可离线工作 — Agent spawn、IM 连接、Skills 执行都在本地，不依赖 Server
  • 离线时 — Memory 召回不可用（降级为无记忆模式），用量数据本地缓存
  • 重连后 — 自动上传缓存的用量数据 + 待写入的记忆条目，Server 补录审计日志
